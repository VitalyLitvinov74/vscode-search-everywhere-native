# Исследование: пользовательский слой над поиском VS Code

Дата среза: 2026-07-21.

## Решение

Проект не создаёт собственный индекс или поисковый движок. Он меняет пользовательский путь и передаёт запросы родным режимам VS Code.

Это решение снимает риски хранения индекса, синхронизации файлов, повреждения данных, упаковки нативных зависимостей и отдельной поддержки Windows, macOS, Linux, WSL, SSH и контейнеров.

## Какой опыт берём у PhpStorm

`Search Everywhere` открывается двойным Shift, использует одну строку запроса и позволяет переключать категории файлов, классов, символов и действий клавишей `Tab`. Отдельный полнотекстовый поиск поддерживает регистр, целые слова, регулярные выражения, маски и области.

Источники:

- [JetBrains: Search Everywhere](https://www.jetbrains.com/help/phpstorm/searching-everywhere.html)
- [JetBrains: Find and replace text in project](https://www.jetbrains.com/help/phpstorm/finding-and-replacing-text-in-project.html)

Нужен не буквальный внешний вид PhpStorm, а короткий путь: открыть, ввести, просмотреть, перейти и при необходимости сузить область.

## Что уже предоставляет VS Code

| Задача | Родная поверхность | Особенности |
|---|---|---|
| Найти файл | `Quick Open`, `Ctrl+P` | Нечёткий поиск, просмотр, открытие рядом |
| Найти файл или символ | `Quick Open` с `search.quickOpen.includeSymbols` | Символы зависят от установленной языковой поддержки |
| Найти символ рабочей области | префикс `#` или команда перехода к символу | Используются поставщики символов VS Code |
| Быстро найти текст | `Quick Search`, префикс `%` | Родной поиск по тексту, просмотр и перенос результата в панель `Search` |
| Выполнить действие | префикс `>` или палитра команд | Используется родной список команд и их названия |
| Выполнить сложный поиск | панель `Search` или редактор поиска | Регистр, целое слово, регулярные выражения, включения, исключения и открытые редакторы |

`Quick Search` стал обычной возможностью VS Code в версии 1.89. Официальные заметки прямо описывают быстрый текстовый поиск по файлам рабочей области и команду `workbench.action.quickTextSearch`: [VS Code 1.89](https://code.visualstudio.com/updates/v1_89).

Официальная документация также описывает `Quick Open`, палитру команд и единое окно с режимами файлов, символов и действий: [интерфейс VS Code](https://code.visualstudio.com/docs/editing/userinterface).

## Что может сделать расширение

### Надёжная основа

- открыть родное быстрое окно через `workbench.action.quickOpen`;
- передать префикс и исходную строку запроса;
- открыть родной `Quick Search`;
- открыть полную панель `Search` с заполненными параметрами через `workbench.action.findInFiles`;
- получить выбранный текст или слово из активного редактора;
- показать собственный небольшой `Quick Pick` для выбора режима, если родного переключения недостаточно;
- хранить только настройки пользовательского пути: последний режим, последнее значение области и выбранное сочетание клавиш.

Исходный код VS Code показывает, что `workbench.action.quickOpen` принимает строку префикса, а `workbench.action.findInFiles` принимает запрос, включения, исключения, регистр, целое слово, регулярное выражение и поиск только по открытым редакторам:

- [`quickAccessActions.ts`](https://github.com/microsoft/vscode/blob/main/src/vs/workbench/browser/actions/quickAccessActions.ts)
- [`searchActionsFind.ts`](https://github.com/microsoft/vscode/blob/main/src/vs/workbench/contrib/search/browser/searchActionsFind.ts)

Эти команды относятся к рабочему интерфейсу VS Code, а не к отдельному поисковому движку. Их нужно изолировать в небольшом слое совместимости и проверять на минимальной поддерживаемой версии VS Code.

### Жёсткое ограничение

Стабильный API расширений не предоставляет `workspace.findTextInFiles`. Он остаётся предлагаемым API, а задача его стабилизации открыта с 2018 года:

- [предложение `findTextInFiles`](https://github.com/microsoft/vscode/blob/main/src/vscode-dts/vscode.proposed.findTextInFiles.d.ts)
- [задача VS Code №59924](https://github.com/microsoft/vscode/issues/59924)

Предлагаемый API нестабилен, доступен в `Insiders` и не должен использоваться в опубликованных расширениях: [правила предлагаемого API](https://code.visualstudio.com/api/advanced-topics/using-proposed-api).

Следствие: расширение не может надёжно получить родные текстовые совпадения и смешать их с файлами, символами и действиями в собственном списке. Попытка разбирать внутреннюю строку `search.action.getSearchResults` или читать DOM будет хрупкой и не входит в основу продукта.

## Предлагаемый пользовательский путь

### Быстрый режим

1. Двойной Shift или настраиваемое сочетание открывает диспетчер.
2. По умолчанию открывается родной поиск файлов и символов.
3. Выбранный текст передаётся как начальный запрос, если это явно включено.
4. Отдельные команды открывают текст, символы, действия и расширенный поиск.
5. Пользователь видит родные результаты, просмотр и поведение открытия.

### Расширенный режим

Для регулярных выражений, масок, исключений и области диспетчер открывает `workbench.action.findInFiles` с заполненными параметрами. Выдачу и дальнейшую навигацию полностью ведёт VS Code.

### Переключение категорий

Идеал — переключать категории клавишей `Tab` без потери запроса. До реализации нужно проверить два варианта:

1. Родные префиксы `#`, `%` и `>` с явными подсказками.
2. Небольшой собственный диспетчер, который хранит запрос до передачи в родную поверхность.

Перехватывать ввод внутри чужого родного `Quick Pick` и читать его значение стабильный API не позволяет. Поэтому обещание бесшовного переключения после передачи управления требует отдельного технического опыта.

## Сигналы пользователей

Открытые обсуждения подтверждают именно проблему разрозненного пути, а не необходимость ещё одного движка:

- Пользователь ищет эквивалент двойного Shift и получает советы отдельно использовать символы и текст: [Stack Overflow](https://stackoverflow.com/questions/76501923/jetbrains-ide-double-shift-to-search-everywhere-equivalent-in-vs-code).
- Пользователь хочет одновременно видеть совпадения по имени файла и его содержимому: [Reddit](https://www.reddit.com/r/vscode/comments/y3syt2/is_there_a_quick_search_featureextension_that/).
- Пользователь после PyCharm описывает выбранный текст, двойной Shift, файлы, действия и символы в одном привычном пути; в обсуждении ему указывают на префикс `%` и сочетание `shift shift`: [Reddit](https://www.reddit.com/r/vscode/comments/1igatrc/switching_from_pycharm_global_search_with/).
- В старом вопросе о мгновенном поиске ответ обновлён после появления родного `Quick Search`: [Stack Overflow](https://stackoverflow.com/questions/45472947/vs-code-quick-search-in-all-files-all-code-instantaneously).

Эти источники являются качественными сигналами, а не статистикой рынка.

## Конкурентная карта после уточнения границы

| Решение | Что делает | Отличие нашей гипотезы |
|---|---|---|
| Встроенный VS Code | Уже имеет файлы, символы, текст, действия и полную панель поиска | Режимы разнесены по командам и префиксам |
| `Search Everywhere` | Собирает файлы, классы, символы и действия, поддерживает двойной Shift | Создаёт собственную индексацию; текст указан в планах |
| `Storm Search` | Даёт двухпанельный поиск в стиле PhpStorm | Запускает отдельный `ripgrep` |
| `Spyglass` | Объединяет несколько источников и просмотр | Имеет собственный слой выполнения поиска |
| `Shift Shift - Search Everywhere` | Объединяет файлы и символы, использует двойной Shift | Не покрывает текст и расширенный поиск |

Наша проверяемая ниша:

> Пользовательский сценарий `Search Everywhere` для VS Code, который не дублирует поиск и передаёт запросы родным режимам редактора.

## Риски

1. Название репозитория с `indexed` больше не соответствует продукту и требует отдельного решения о переименовании.
2. Истинный единый смешанный список невозможен на стабильном API без собственного поиска.
3. Команды рабочего интерфейса могут меняться между версиями VS Code; нужен слой совместимости и проверка версий.
4. Двойной Shift может конфликтовать с раскладкой, специальными возможностями и другими расширениями; обязателен запасной настраиваемый вызов.
5. Полнота символов зависит от языковых расширений.
6. Родной `Quick Search` не заменяет расширенную панель для масок и регулярных выражений.
7. Расширение рискует быть слишком тонким; ценность нужно доказать измерением числа действий и времени.

## Рекомендации

1. Прототипировать только диспетчер родных режимов.
2. Не изменять пользовательские настройки автоматически.
3. Сравнить три входа: сразу файлы и символы; меню режимов; восстановление последнего режима.
4. Проверить передачу выделенного текста, открытие рядом и переход в полную панель.
5. Измерять путь пользователя, а не скорость поискового движка.
6. После проверки выбрать новое описательное имя без слова `indexed`.
