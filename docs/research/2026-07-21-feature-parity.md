# Матрица паритета поиска PhpStorm и VS Code

Дата среза: 2026-07-21.

## Назначение

Матрица переводит исследование рынка и поведения PhpStorm в проверяемые требования. Она не требует копировать продукт JetBrains буквально: каждая возможность либо воспроизводится, либо адаптируется к ограничениям стабильного API VS Code, либо явно откладывается.

## Обозначения

- `P0` — исследование или архитектурное решение до начала основной реализации.
- `P1` — обязательное поведение первой публичной версии.
- `P2` — следующий этап после подтверждения первой версии.
- `P3` — дальнее развитие.
- `OUT` — намеренно не входит в обозримый объём.

Тип решения:

- **Паритет** — воспроизводим узнаваемое поведение PhpStorm.
- **Адаптация** — сохраняем пользовательскую цель, но используем возможности VS Code.
- **Собственное** — добавляем необходимое для надёжного постоянного индекса.

## Кластеры подтверждённых проблем

Обсуждения сообщества используются как сигналы симптомов, а не как доказательство размера рынка.

| Кластер | Наблюдаемый симптом | Доказательства | Уверенность | Следствие |
|---|---|---|---|---|
| Разрозненный вход | Для файлов, текста, символов и команд используются разные окна и сочетания клавиш | [Reddit: аналог Search Everywhere](https://www.reddit.com/r/vscode/comments/hab1k1), [Search Everywhere PhpStorm](https://www.jetbrains.com/help/phpstorm/searching-everywhere.html) | Средняя | Единый вход и переключение категорий обязательны |
| Повторная задержка | Пользователи больших и удалённых проектов хотят предварительное индексирование вместо полного сканирования на каждый запрос | [Stack Overflow: ускорение поиска](https://stackoverflow.com/questions/40611840/how-can-i-make-vs-codes-search-in-files-go-faster/76957600), [Stack Overflow: индексирование удалённой папки](https://stackoverflow.com/questions/78075380/how-to-tell-vscode-to-index-the-entire-project-folder) | Средняя | Постоянный индекс содержимого — основа продукта |
| Повторное сканирование | Даже при постоянной базе перечисление большого числа файлов после запуска воспринимается как повторная индексация | [vscode-cpptools #13927](https://github.com/microsoft/vscode-cpptools/issues/13927) | Средняя | Повторный запуск должен иметь измеримый быстрый путь |
| Дорогая или циклическая индексация | Пользователи PhpStorm сталкиваются с повторными пересборками, высокой нагрузкой и повреждёнными индексами | [Stack Overflow: continuous indexing](https://stackoverflow.com/questions/78400438/continuous-indexing-issue-in-phpstorm), [JetBrains: Repair IDE](https://www.jetbrains.com/help/phpstorm/repair-ide.html) | Средняя | Нужны пауза, восстановление, версия формата и понятное состояние |
| Исключения и области | Пользователи требуют учитывать `.gitignore`, маски файлов и запоминать область между поисками | [Storm Search #2](https://github.com/zigcBenx/storm-search/issues/2), [#35](https://github.com/zigcBenx/storm-search/issues/35), [#37](https://github.com/zigcBenx/storm-search/issues/37) | Высокая для UX существующего расширения | Маски, исключения и сохранение состояния входят в первую версию |
| Клавиатурная навигация | Нельзя считать интерфейс «как в PhpStorm», если фокус, просмотр и открытие результата требуют мыши | [Storm Search #4](https://github.com/zigcBenx/storm-search/issues/4), [Reddit: Storm Search](https://www.reddit.com/r/vscode/comments/1p3s4gj/i_forked_vs_code_to_make_global_search_actually/) | Средняя | Все основные действия должны иметь клавиатурный путь |
| Качество просмотра | Пользователи ожидают подсветку синтаксиса, тему редактора, перенос строк и ограниченный контекст | [Storm Search #14](https://github.com/zigcBenx/storm-search/issues/14), [#18](https://github.com/zigcBenx/storm-search/issues/18), [Spyglass #3](https://github.com/garroter/spyglass/issues/3) | Средняя | Просмотр результата является частью поиска, а не декоративным дополнением |
| Сохранение контекста | После открытия неверного результата пользователь хочет вернуться к тому же запросу, маске и области | [Storm Search #31](https://github.com/zigcBenx/storm-search/issues/31), [#37](https://github.com/zigcBenx/storm-search/issues/37) | Высокая для UX существующего расширения | Состояние поиска и история не должны сбрасываться |
| Конфиденциальность | Для расширения, читающего весь проект, пользователи ожидают явное описание собираемых данных и разрешений | [Spyglass #1](https://github.com/garroter/spyglass/issues/1) | Средняя | Локальная обработка и отсутствие сетевой передачи должны быть проверяемым свойством |

## Архитектурные проверки `P0`

До выбора движка нужно закрыть следующие вопросы:

| Код | Проверка | Решение считается подтверждённым, если |
|---|---|---|
| `SPIKE-ENGINE` | Сравнить `SQLite FTS5` через WebAssembly и специализированный триграммный индекс | Один набор корпусов и запросов даёт воспроизводимые данные по времени, памяти, размеру, точности и обновлению |
| `SPIKE-THREAD` | Проверить работу вне основного потока процесса расширений | Индексирование и запросы не создают заметных зависаний интерфейса |

| `SPIKE-WATCH` | Проверить наблюдатели и сверку | Создание, изменение, переименование и удаление каталогов не оставляют устойчиво устаревшие записи |
| `SPIKE-RECOVERY` | Принудительно завершить индексирование и повредить индекс | Следующий запуск восстанавливается или безопасно перестраивает данные |
| `SPIKE-QUERY` | Проверить короткие запросы, Unicode и регулярные выражения | Для каждого класса запроса известны план, ограничения и точная проверка |

## Границы первой публичной версии

Первая публичная версия должна закрыть все строки `P1`. Ключевой пользовательский путь:

1. Пользователь открывает большую рабочую область.
2. Состояние индекса видно, но поиск доступен сразу.
3. Двойной Shift открывает единое окно с недавними файлами.
4. Пользователь вводит запрос и получает файлы, символы, текст и действия.
5. Tab меняет категорию без потери запроса.
6. Маска и область сужают выдачу.
7. Стрелки меняют результат и полный просмотр; Enter открывает, Ctrl+Enter открывает рядом.
8. После изменения или несохранённой правки результат обновляется без полной пересборки.
9. После перезапуска поиск открывается на готовом индексе и сохраняет предыдущий контекст.

## Не входит в первую публичную версию

- массовая замена;
- структурный поиск и замена;
- самостоятельный разбор всех языков;
- поиск использований и граф вызовов;
- семантический поиск;
- облачная синхронизация;
- общие индексы между компьютерами;
- история Git и поиск по нескольким веткам;
- браузерная версия расширения.

## Предварительные измеримые критерии

Пороги должны быть подтверждены технической проверкой:

- первая страница буквального запроса длиной от трёх символов: `p95 ≤ 100 мс` после прогрева;
- изменение сохранённого файла отражается в выдаче: `p95 ≤ 1 с`;
- повторный запуск неизменённой рабочей области не читает содержимое всех файлов;
- индекс занимает не более `3×` объёма индексируемого текста;
- старый запрос отменяется после изменения строки поиска;
- поиск работает во время первичного построения;
- принудительное завершение не оставляет индекс в неустранимом состоянии;
- исходный код и поисковые запросы не отправляются в сеть.

## Открытые продуктовые вопросы

1. Подтвердят ли пользователи ценность единого окна выше, чем ценность отдельной быстрой панели текста.
2. Обязательны ли регулярные выражения для первого выпуска или их допустимо включить в следующий этап.
3. Должен ли режим «Все» показывать текст сразу или, как PhpStorm, опускать его ниже файлов, символов и действий.
4. Следует ли запоминать последний режим или каждый раз открывать общий режим с недавними файлами.
5. Насколько рано нужна совместимость с Cursor и другими редакторами на основе VS Code.
6. Какой предел размера файла и общего объёма рабочей области должен действовать по умолчанию.
